# 【開発仕様・引き継ぎ資料】対話型教育システムにおける「発問データベース」設計指針

> 元資料（背景・①〜④・階段）に、Claude による追補（⑤〜⑯・階段の補足・テンプレ）を統合した完全版。

## 1. 開発の背景とコア・バリュー
本システムは、オフライン環境であっても、生徒が「一流の教師と対話しているような体験」を得られるEdTechソフトである。
従来のAIドリルが「問題の分析（結果のデータ）」に終始しているのに対し、本システムは**「発問に対する生徒の反応（つまずきのプロセスデータ）」をデータベース化・構造化する**点で、既存製品と圧倒的な差別化を図る。

---

## 2. 「いい発問」の定義と4つのコア・ルール
本システムにおける「いい発問」とは、**肝心な答えや公式は教師（システム）側からは言わず、生徒が自力で次のステップに上がるための問い**を指す。定義や暗記要素は普通に説明し、思考を促す場面で以下の4つのルールを適用する。

> ※①〜④は「**何を問うか＝発問のフック（中身）の4類型**」。これに対し、後述の⑤〜⑯は「**どう問い・どう運び・どう締め・何を記録するか**」という別の軸の条件である。両方そろって初めて機能する。

### ① 【目的・願望】の発問（「嬉しい？」を引き出す）
* **思想:** 単に「解きなさい」と命じるのではなく、「どうなったらラクか、嬉しいか」という生徒の主観・願望に接続する。
* **例（方程式）:** `$3x = 12$` で「両辺をどうする？」と聞くのではなく、「ここが『ただの $x$』だったら一瞬で答えが分かって嬉しいんだけど、何だったらいい？」と問いかける。

### ② 【アナロジー（類推）】の発問（既習事項を武器にする）
* **思想:** 新しい知識を孤立させず、「過去の貯金（既に知っていること）」の延長線上に位置づける。
* **例（文字の式）:** `$3x + 2x$` を教える際、「中身は見えないけど同じ重さのリンゴの箱 $x$ が左に3箱、右に2箱あります。合わせて何箱？」と問い、小学校の足し算と同じだと気づかせる。

### ③ 【揺さぶり（認知葛藤）】の発問（「あれ？使えない！」を演出する）
* **思想:** あえて直感や前の知識が通用しない罠（バグ）を提示し、新しい概念を学ぶ必要性を肌で感じさせる。
* **例（比例・反比例）:** 反比例 `$y = \frac{6}{x}$` の導入時、「比例のグラフは点を2個打って定規で引けば一瞬だったよね。じゃあ、これも点を2個だけ打って定規でピッと引いちゃっていいと思う？」と問いかける。

### ④ 【感動・アハ体験】の発問（「なるほど！」を仕込む）
* **思想:** 「伏線の回収」「直感の破壊」「世界の翻訳（日常や心理との接続）」を仕込み、数学の美しさや合理性に鳥肌が立つ瞬間を作る。
* **例（正負の数の減法）:** `$-5 - (-3)$` の意味を、「自分の中の嫌いなところ（マイナス）を3個、自分の手で引き算（消去）したら、君の心はプラスとマイナスどっちに近づく？」と言語化し、意味を腑に落ちさせる。

## 3. 「いい発問」の条件・追補（⑤〜⑯）
①〜④（中身）に加えて、すべての発問・対話が満たすべき条件。**A 性質 / B 運び方 / C 締め / D 土台 / E データ化**の5軸。

### A. 発問の"性質"の条件

**⑤【低い入口・高い天井】**
* **思想:** 最初の一歩は誰でも乗れるようにする。考えられるのは得意な子だけ、にしない。入口は易しく、その先は深く。
* **例:** いきなり「式は？」ではなく「数えてもいい、まず何本になりそう？」——全員が必ず何か言える入口にする。

**⑥【焦点はひとつ・一度にひとつ】**
* **思想:** 「何か気づく？」は漠然としすぎ。視線を一点に集める。問いを重ねて出さない。
* **例:** 「左の式と右の式、"どこが"違う？」と、見る場所を指定する。

**⑦【半歩先・ペースは可変（"飛ばす"足場）】**
* **思想:** 刻みは「今この子のちょうど半歩先」。**先に進んでいる子には段を飛ばす**。刻みすぎは"急かし"であると同時に冗長で非効率。足場は降りるだけでなく**昇る**（具体→一般化）方向も持つ。
* **実装:** 発問の木に「**飛ばし条件**（この答えが出たら一気に次の山場へジャンプ）」を持たせる。

### B. 対話の"運び方"の条件

**⑧【待つ】沈黙を急いで埋めない**
* **思想:** 答えが出ないとき、すぐ補助発問を出さない。「考える時間とるね」と数秒待つ。沈黙＝思考中。これが抜けると、せっかくの発問が"急かし"になる。

**⑨【誤答を燃料にする】間違いを正さず、前に進める**
* **思想:** 「違うよ」で終わらせない。「**その考えでいくと、ここはどうなる？**」と、間違いごと一歩進める。誤答は捨てるものでなく、次の発問の材料。

**⑩【正解で止めない】理由と一般化を問う**
* **思想:** 当たっても「正解！」で終わらない。「**なぜそう言える？**」「**数が変わっても言える？**」。③揺さぶりが"導入のフック"なのに対し、これは"正解の直後"に使う深掘り。
* **例:** 16本を数えで当てた子に「じゃあ100個でも数える？」——正解を一般化へ押し出す。

**⑪【言語化させる】"どうやって？"を必ず聞く**
* **思想:** 答えだけでなく方法を言わせる。「どうやって出した？」「頭の中で何をした？」。言語化した瞬間に、"なんとなく"が"使える知識"に変わる。

**⑫【生徒の方法を採用する】教科書の解き方を押しつけない**
* **思想:** 生徒が自分のやり方（例：`$3×100+1$`）を出したら、それを黒板に乗せて育てる。"先生の正解ルート"に強制連行しない。所有感が学習を深める。

### C. "締め"の条件

**⑬【ふりかえらせる】"結局、何がわかった？"**
* **思想:** 最後に生徒自身の言葉でまとめさせる。「どこで気づいた？」。これが次の授業の"既習の貯金"（②アナロジー）になる。

**⑭【次の問いを生徒に持たせる】**
* **思想:** 最上位は、生徒が次の問いを持つこと。「じゃあ次、何が気になる？」。問いを"もらう"側から"つくる"側へ。

### D. 土台（これが無いと①〜⑭が機能しない）

**⑮【安心して間違えられる空気】評価でジャッジしない**
* **思想:** 4類型も足場も、"間違えても大丈夫"が無いと動かない。否定しない・急かさない・正誤で値踏みしない。発問の前提条件。

### E. DB設計への追加（"プロセスデータでの差別化"を効かせる核）

**⑯【想定誤答 × その背後の誤概念(misconception) をタグ付け】**
* **思想:** 補助発問を出すだけでなく、「この誤答が出た＝この誤概念」を記録する。これが後で"診断"と"個別最適"の素になる。
* **連結:** 後述の階段ステップ3「二者択一（4択）」の選択肢は、この誤概念から作る——ただの当てっこでなく、「なぜそれを選んだ？」に繋がる診断装置にする。

## 4. スモールステップ（補助発問）の「階段の幅」
メイン発問に生徒が答えられなかった、あるいはフリーズ（沈黙）した場合、システムは以下の順に**「思考の解像度を1段階ずつ下げる（具体化する）」**補助発問を繰り出す。

```text
【メイン発問：ジャンプ（主観・願い）】
   └── 躓き ──> 【ステップ1：既習との比較】「前の時間の〇〇とどこが似てる？」
                  └── 躓き ──> 【ステップ2：数値の具体化・視覚化】「数直線（図）にするとどうなる？」「数字を2に変えたら？」
                                 └── 躓き ──> 【ステップ3：二者択一／4択】「増えると思う？減ると思う？」
                                                └── 躓き ──> 【最終：説明する】定義・約束・結論は問わずに普通に教える
```

**補足（⑤〜⑯を反映）:**
- **降りるだけでなく昇る:** ステップで具体に降ろした後は、必ず元の抽象（一般化）へ**登り直す**問いをかける（具体⇄抽象の往復）。
- **ステップ3の選択肢は誤概念ベース:** 4択の誤答は⑯の「想定誤答」から作り、選んだら「なぜ？」を問う。
- **降りる前に"待つ"（⑧）:** 躓き＝即・補助発問、ではない。まず数秒待つ。
- **飛ばし条件（⑦）:** スムーズな子は階段に入れず、メイン発問だけで次の山場へ送る。

## 5. 発問の木（テンプレート）
1授業＝「中心発問の背骨」＋「躓いたら差し込む補助発問の枝」。各ノードに以下を持たせる。

| 項目 | 中身 |
|---|---|
| 中心発問 | その山場の問い。答え方（自由記述／手書き／4択）を指定 |
| ねらい | この発問で到達させたいこと |
| 板書 | この発問で黒板に残すもの |
| 飛ばし条件（⑦） | この答えが出たら次の山場へジャンプ |
| 想定誤答×誤概念（⑯） | 出やすい誤答と、その背後の誤概念 |
| 補助発問（階段） | 躓いたとき上から降ろす。ステップ1→2→3→説明 |

具体例は `発問の木_マッチ棒.md` を参照。
